summary

Conducted a user research on the influence of social comparison on task performance and developed an extension of the game Lunar Lander on the android phone.





background

In most multiplayer games, the leaderboard can be used as an indicator to see the performances of players. A leaderboard can show score, the amount of kills ,etc. It can let one know that they are improving, getting worse, or just unlucky. The leaderboard can let you know if you are against good or bad players. And sometimes that can influence the way you play. In this case study, I wanted to research if players were affected by looking at their own or other player’s stats.


what i did

  • Conducted a series of surveys to gather and collect data on participants to determine external factors that may affect performance.
  • Extended an existing Android implementation of the game Lunar Lander to collect quantitive data.
  • Redesigned the game's basic game mechanics to accomodate for the comparative evaluation between two different interfaces.
  • Designed and updated game's interface and design to add a more aethestically pleasing experience and game flow to fit the experiment.


pre-experiment questionnaire

A questionnaire was set up for the experiment to gather information that may affect performance. Factors such as time on computer, internet, and mobile were all considered and observed.



start + setup

To make the experience more game-like, I redesigned the interface into a game menu and improved the overall user flow. Rather than hand the game over to participants after inputting the values to be recorded, I wanted the user to feel like they are playing a game and not feel like they are partaking in an experiment as to not influence their results. I would add a game menu to the original app that the user can select ‘PLAY’ or ‘QUIT’ from, and the ‘SETUP’ would serve as an option meant for researchers.

To add from the original app, I would also try making the game from a locked portrait view to a locked landscape. This is give the game more landing spaces and allow the user to have more interesting gameplay.









PLAY loads the player into the game with all the participant data via ‘SETUP’.

SETUP loads the setup page and allows inputs to be adjusted by the researcher based on the trial. Once all settings for setup are saved, exiting will return back to the start menu.

QUIT simply exits the app.

PARTICIPANT CODE is the participant number.

SESSION CODE is the number for the session.

GROUP CODE refers to the two study groups: leaderboard vs. non.

LIVES is the number of attempts each user gets if they fail a landing. The default is 3 and is what is being used for the experiment. The max is 10, min is 1.

LEADERBOARD can set to on or off.




game rules + controls





Pressing ‘PLAY’ loads the participants into the game where they can begin scoring points. Successful landing on marked areas with _X will score the player points based on the landing location. Successfully landing a lunar lander is determined by which angle the lander falls. If the angle is perfectly orthogonal, the player will score perfect points. Angles that are between 45-90 degrees of the lander will count as a successful land and count as scoring points. Angles least close to 90 degrees will have deducted score. Successful landings will result in the start of a new level and increased difficulty. Angles less than 45 degrees will count as unsuccessful landings. Unsuccessful landings decrement lives by 1. Scores are also influenced by the amount of time and remaining fuel.




Overall calculations for scores are as follows:

Score = (2 * lander angle) + 250
Score = 5 * (100 / completion time in seconds)
Score = 5 * (remaining fuel)

Numerical values for fuel ranges from 0 to 100, with fuel starting off at 100. Time is calculated in seconds. Speed is calculated using the built-in source code’s vector physics. Increased level difficulties are calculated below with these values in mind.





leaderboard




Selecting the menu option on the top left will pop up the options for RESUME, LEADERBOARD, or QUIT. Depending on the setup some players will not see the option for LEADERBOARD.




For players with the LEADERBOARD option ON, participants will see the top 5 scores between other participants or a set score (if the participants performing the experiment are the first 5).

QUIT will simply just exit the game back to the game menu. If the player quits the game before the number of lives run out, data will not be recorded.


game over + data collection

The game is over when the participant’s lives reaches zero. Quitting early does not result in a data save and information is only saved when the number of remaining lives reaches 0. Data is saved via an app called Astro (via .txt files) and values are sent as Average level before dying, average time per level, average fuel per level, highest level reached, and highest score reached.







conclusion

In this user study, social comparison had improvement on scores and completion times for participants with a leaderboard. However, little improvement was observed from leaderboard revelation and without. Participants showed interest in learning the game suggesting that the game was hard, or the participants gathered were not frequent gamers.


future work

Future work includes modifying the pre-experiment interview to gather information on participants' gaming history as frequent gamers show more competitiveness than inexperienced players.
